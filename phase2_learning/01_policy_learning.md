# Policy Learning
### Phase 2 — Learning Layer, Doc 01

*Part of [Transition Grammar for Reasoning Systems](../../README.md)*

---

## Status

This doc is a structured stub. Policy learning is the third control loop defined in Doc 02 — the mechanism by which operator selection improves over time from accumulated Δ-gap data. The direction is clear; the implementation requires the logging infrastructure from `/phase2_implementation/` to be running first.

This doc defines the goal, the signal, and the open problems. Implementation follows from data.

---

## The Goal

Replace hand-assigned weights and heuristic operator selection with a learned policy that minimizes Δ-gap over time.

Current state: weights in the cost function are fixed at v0.1 defaults. The weight controller `f` adjusts them heuristically based on pressure score. Neither is learned.

Target state: a policy `π(s) → operator` that selects transitions with consistently lower Δ-gap than heuristic selection, trained from logged transition data.

---

## The Signal

Δ-gap is the reward signal:

```
R = −Δ_gap
```

Low Δ-gap = the system's prediction was accurate and the selected transition was structurally sound. High Δ-gap = the system chose worse than it could have, or predicted badly.

This is process-level supervision — rewarding trajectory quality, not output correctness. A response that looks helpful but collapses under the next challenge produces high Δ-gap. A response that admits uncertainty and holds the state produces low Δ-gap.

---

## What Gets Learned

Three things, in rough priority order:

**1. Δ predictor calibration**: Systematic biases in `g` (over-predicting ρ for DEFER, under-predicting m for REFRAME) are correctable from logs. Adjust predictor calibration before adjusting weights.

**2. Weight controller `f`**: Which pressure levels warrant which weight adjustments. Currently hand-tuned (α, β constants). Log analysis reveals where the hand-tuning is wrong.

**3. Operator selection policy**: Given state `s` and instability type, which operator is most likely to produce low Δ-gap? This is the full policy — requires enough logged data to be meaningful (rough estimate: 500+ transitions before patterns are reliable).

---

## Open Problems

Three problems remain unsolved and will not be solved by logging alone:

**Bootstrap problem**: The policy learns from logged Δ-gap data. But the initial logs are generated by a heuristic policy, not a learned one. Early data may be systematically biased toward whatever the heuristic prefers. Breaking this circle requires either synthetic data, diverse prompt coverage, or explicit exploration.

**Exploration vs. safety**: A learning policy needs to try suboptimal options to discover their actual Δ. But trying suboptimal transitions in live use causes real failures. Safe exploration — choosing slightly suboptimal candidates in low-stakes contexts while staying conservative in high-stakes ones — is unsolved at v0.1.

**Optimal transition undefined**: The policy minimizes Δ-gap relative to Δ̂_ref, which is the best predicted candidate. If the candidate set is poor (Candidate Set Collapse), minimizing Δ-gap within a bad set doesn't help. Policy learning cannot compensate for bad candidate generation.

---

## Minimal Next Step

Before any policy learning:

1. Run the Glimmer-Gate wrapper for 200+ turns across varied prompt types
2. Compute Δ-gap statistics by operator and instability type
3. Identify systematic biases (which operators consistently overperform or underperform their predictions)
4. Manually adjust predictor calibration for the worst-performing operators
5. Repeat

This manual loop is policy learning without automation. It's sufficient to validate that the signal is real before investing in learned policies.
